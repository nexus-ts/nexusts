# Internationalization · `@nexusts/i18n` (v0.5)

> New in v0.5. Locale-aware translation and formatting for the
> Bun-native stack. Zero external dependencies — uses the built-in `Intl` API.
> built-in `Intl` API.

`@nexusts/i18n` provides:

- **`I18nService`** — translates messages, formats dates / numbers
  / currency via `Intl`.
- **`I18nModule.forRoot(config)`** — wires the service into the
  DI container, optionally loads JSON files from a directory.
- **`i18nMiddleware(service)`** — Hono middleware that detects the
  active locale per request from `?lang=` → `lang` cookie →
  `Accept-Language` → default.
- **`@CurrentLocale()`** — controller parameter decorator that
  injects the active locale string.

---

## 1. Quick start

```ts
import { Module, Inject } from "@nexusts/core";
import { I18nModule, I18nService, I18N_SERVICE_TOKEN, CurrentLocale } from "@nexusts/i18n";

@Module({
  imports: [
    I18nModule.forRoot({
      defaultLocale: "en",
      messages: {
        en: {
          hello: "Hello, :name!",
          "auth.errors.invalid": "Invalid email or password",
        },
        ko: {
          hello: "안녕하세요, :name님!",
          "auth.errors.invalid": "이메일 또는 비밀번호가 올바르지 않습니다",
        },
      },
    }),
  ],
})
class AppModule {}

@Injectable()
class UserService {
  @Inject(I18N_SERVICE_TOKEN) declare i18n: I18nService;

  greet(name: string, locale: string) {
    return this.i18n.t("hello", { name }, locale);
  }
}

@Controller("/users")
class UserController {
  @Get("/")
  index(@CurrentLocale() locale: string) {
    return { greeting: "Welcome", locale };
  }
}
```

## 2. The `t()` function

```ts
i18n.t("hello", { name: "Alice" });            // → "Hello, Alice!" (default locale)
i18n.t("hello", { name: "Alice" }, "ko");     // → "안녕하세요, Alice님!"
i18n.t("missing.key");                         // → "missing.key" (raw key, so the dev sees it)
i18n.tOr("missing", "fallback text");          // → "fallback text"
```

### Interpolation

Use `:name`-style placeholders. Unknown placeholders are left
as-is (so the dev sees `:missing` in the rendered output).

```ts
"profile.welcome": "Welcome, :name! You have :count notifications."
i18n.t("profile.welcome", { name: "Alice", count: 5 });
// → "Welcome, Alice! You have 5 notifications."
```

### Pluralization with `|`

The `|` character splits a message into plural forms. The number
of segments maps to CLDR plural categories:

| Segments | Categories (in order) |
| -------- | --------------------- |
| 1 | other |
| 2 | one \| other |
| 3 | zero \| one \| other |
| 4 | zero \| one \| two \| other |
| 5 | zero \| one \| two \| few \| other |
| 6 | zero \| one \| two \| few \| many \| other |

```ts
// English: 0 → "other", 1 → "one", 2+ → "other"
// So 2-segment form covers all cases:
"items.count": "1 item|:count items"

i18n.t("items.count", { count: 0 });  // → "0 items"
i18n.t("items.count", { count: 1 });  // → "1 item"
i18n.t("items.count", { count: 5 });  // → "5 items"
```

The framework uses `Intl.PluralRules` under the hood, so language
pluralization is automatic — `one` for English 1, `other` for
everything else, and locale-specific rules for Slavic / Arabic /
Welsh / etc.

`Intl.PluralRules` is **not** the same as `Intl.NumberFormat`'s
default. For example, Korean doesn't distinguish plural, so
`Intl.PluralRules("ko").select(0)` returns `"other"` — your
Korean messages should use the 2-segment form `1\uac1c|:count\uac1c`.

### Nested keys

Use dot notation to traverse the message dict:

```ts
messages: {
  en: {
    auth: {
      welcome: "Welcome, :name!",
      errors: { invalid: "Invalid credentials" },
    },
  },
}

i18n.t("auth.welcome", { name: "Alice" });        // → "Welcome, Alice!"
i18n.t("auth.errors.invalid");                    // → "Invalid credentials"
```

## 3. Locale fallback

When a key isn't found in the requested locale, the service
falls back to:

1. The same locale with the region stripped (`fr-CA` → `fr`)
2. The default locale (configurable)
3. The key itself (dev-visible)

```ts
const i18n = new I18nService({
  defaultLocale: "en",
  messages: {
    en: { hi: "hi" },
    fr: { hi: "salut" },
  },
});

i18n.t("hi", undefined, "fr-CA");  // → "salut" (region fallback)
i18n.t("hi", undefined, "ko");    // → "hi"   (default fallback)
i18n.t("missing");                  // → "missing"
```

You can disable the default-locale fallback if you want strict
locale enforcement:

```ts
const i18n = new I18nService({ fallbackToDefault: false });
i18n.t("hi", undefined, "ko");     // → "hi"  (the key, since "ko" has no "hi")
```

## 4. Locale detection (the middleware)

The middleware is installed in your Hono app. It extracts the
locale per request and stores it in `c.var.locale`.

```ts
import { Hono } from "hono";
import { i18nMiddleware, I18nService } from "@nexusts/i18n";

const svc = new I18nService({ /* ... */ });
const app = new Hono();
app.use("*", i18nMiddleware(svc));
```

Priority (first match wins):

1. **Query string** — `?lang=ko` (key configurable)
2. **Cookie** — `lang=ko` (key configurable)
3. **`Accept-Language` header** — parsed with quality scores
4. **Default locale**

The middleware also attaches the `I18nService` instance to
`c.var.i18n` for convenience.

## 5. Formatters

All formatters use Node's built-in `Intl` API and cache their
`Intl.*` instances.

### Dates

```ts
i18n.formatDate(new Date());                 // default locale
i18n.formatDate(new Date(), { locale: "ko-KR" });
i18n.formatDate(new Date(), {
  locale: "en-US",
  year: "numeric", month: "long", day: "numeric",
});
// → "June 22, 2026"
```

### Numbers

```ts
i18n.formatNumber(1234.56);                  // "1,234.56" (en-US)
i18n.formatNumber(1234.56, { locale: "de-DE" });  // "1.234,56"
```

### Currency

```ts
i18n.formatCurrency(1234.56, { currency: "USD" });            // "$1,234.56"
i18n.formatCurrency(1234567, { currency: "KRW", locale: "ko-KR" });  // "₩1,234,567"
```

### Locale-aware compare

```ts
const a = ["B", "a", "C"];
const sorted = [...a].sort((x, y) => i18n.compare(x, y));
// ['a', 'B', 'C'] — case-insensitive sort
```

## 6. Loading messages from disk

Two options:

**Option A**: pass messages inline

```ts
I18nModule.forRoot({ messages: { en: {...}, ko: {...} } });
```

**Option B**: load from a directory of `*.json` files (Node only)

```
locales/
  en.json    { "hello": "Hello, :name!" }
  ko.json    { "hello": "안녕하세요, :name님!" }
  fr.json    { "hello": "Bonjour, :name !" }
```

```ts
I18nModule.forRoot({
  defaultLocale: "en",
  messagesDir: "./locales",
});
```

Each file's filename (without `.json`) becomes the locale code.

For Cloudflare Workers / other edge runtimes, the `fs` module
isn't available — use **Option A** and bundle your messages at
build time.

## 7. Adding messages at runtime

```ts
const svc = container.resolve(I18nService);

// Add a new locale
svc.addMessages("fr", { hello: "Bonjour, :name !" });

// Add (and merge into existing) — useful for plugins
svc.addMessages("en", { plugin: { greeting: "Hi from plugin" } });
```

`addMessages` recursively merges nested dicts, so you can extend
a locale incrementally.

## 8. Configuration

```ts
interface I18nConfig {
  defaultLocale?: Locale;                  // default: "en"
  fallback?: boolean;                     // default: true (region → default)
  fallbackToDefault?: boolean;            // default: true
  supportedLocales?: Locale[];            // empty = all allowed
  messages?: MessageCatalog;              // inline
  messagesDir?: string;                   // path to *.json files
  detectQueryKey?: string;                // default: "lang"
  detectCookieKey?: string;               // default: "lang"
}
```
